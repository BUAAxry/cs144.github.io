Lab FAQ

General

  1. How will code style be graded?

    2. Complex pieces of code should be well documented such that they become easy for the reader to understand (but code should not be over-commented). Common, general-purpose components should be decomposed into helper functions to avoid repeated code, but the code should not be over-decomposed to the point that it is difficult to actually understand the flow of execution. Code should be generally readable, use consistent naming conventions, and not contain commented-out code. Please check the return values of all system calls or functions in which an error could occur, and handle such errors gracefully. Your style grade also includes memory errors and leaks, egregious performance issues (e.g. allocating large amounts of memory for no reason) and unnecessarily complex approaches (e.g. implementing a complicated data structure that is only marginally more efficient than a linked list). Please run make format before submitting each assignment and remove any dummy code or comments that were included with the starter code.

  2. How much of my grade will come from style and to the quality of my writeup?

    3. 15%. The remaining 85% comes from functionality.

  3. How do I run an individual test?

    4. ctest -R '^test_name$'

  4. Can I add include lines in the header files?

    5. Sure.

  5. Why does 'make check' fail even though it's in the Sponge README?

    6. Short Answer: This is a typo in our documentation. You should run make check_lab0, not make check.
    Long Answer: The make target, check, runs the entire sponge test suite. The full test suite requires a shell script, tun.sh, which you won't get until a later lab. The make target check_lab0 will just test lab0. You should be able to run it with the lab0 starter code (although it will return an error until you've completed the lab!).

  6. What's the procedure if I want to use some of my late days?

    7. You must email the staff list by the lab deadline letting us know if you are using late days. With weekly labs, the CAs want to be able to get feedback to you quickly. As such, we will start grading as soon as the deadline passes, and want to avoid redundantly grading submissions that weren't intended to be final.

Lab 0

  1. What should the behaviour of my program be if the caller tries to peek or pop with a len greater than what is available?
    2. We don't have any preference on the behavior if the caller tries to peek or pop more than is available in the stream, as long as you behave reasonably and don't crash in that situation. If you want to peek/pop the maximum available, that is fine with us. If you want to throw an exception (either for an overlength peek or an overlength pop), that is also fine with us.

Lab 1

  1. What is the definition of capacity?
    2. Capacity is the maximum sum of the number of bytes that have been assembled but not yet read and those that are unassembled. For example, if the capacity is 10 bytes, there have been 7 bytes assembled and pushed to the output stream, 4 of which have been read, and there are 5 bytes unassembled, then remaining capacity is 10 - 3 (assembled but not read) - 5 (unassembled) = 2.
